Science and the decentralized web - Managing signature collections using dat

02 Aug 2018

dat is pretty exciting because the main use case, easily sharing files with other people, fits very well in a scientific workflow. On top of that, the Beaker browser is a nice demo of what a browser with dat built-in looks like, and it's pretty cool to have easy forking and editing of websites (with integrated versioning and everything).

For this PoC I want to use the same UI for drag and dropping sequence data and calculating the signatures, but allow the user to save the signatures in a dat archive. This is similar to how Fritter and Rotonde save the user profiles, and it's nice because it allows a separation between the UI and the data (so when new updates to the UI are available, the data still works just fine). The alternative would be to put everything in the same dat archive, but then users would need to 'upgrade' the dat archive themselves when new UI versions are available.

How did it go?

My initial plan was to run it all thru Beaker, but due to wasm-bindgen using the BigInt JS spec and electron 2.x (a Beaker dependency) being on Chromium 61 I can't test it there yet... 1

Trying to do something on a regular browser it is, then. Turns out that dat-js is out of date and there is no clear way to use dat (there are suggestions for using hyperdrive directly, but it needs a dat-gateway to be able to access the network, which is clumsy and kind of defeats the purpose of the no-server part). There is a good demo showing how to do that, but at this point I think I'll let the idea floating some more and wait for a newer Chromium on Beaker.
